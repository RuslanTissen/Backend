## Погружение в REST / REST API

# Что такое REST?

- В значительной степени опирается на основы HTTP (HTTP — это стандартный язык (протокол), на котором браузер просит данные у сервера, а сервер их передает).
- REST (Representational State Transfer) — это не протокол, а архитектурный стиль. Это набор правил о том, как сервер и клиент должны общаться друг с другом, чтобы система была масштабируемой и понятной.
- (https://www.codecademy.com/articles/what-is-rest)
- (https://restfulapi.net/)

## Основные принципы:

- Stateless (Без сохранения состояния): Сервер не помнит, что вы спрашивали секунду назад. Каждый новый запрос должен содержать абсолютно всю информацию, необходимую для его выполнения (all relevant information is in the Req/Res).

- Разделение Клиента и Сервера: Серверу плевать на ваш интерфейс, а Клиенту плевать, как данные хранятся в базе. Они общаются через стандартизированный интерфейс.

- Ресурсы: В REST мы работаем с «ресурсами» (User, TODO Item, Post, Comment...), а не просто вызываем функции.
- Clients make requests to the server to get or modify data on the server.

## Анатомия запроса

# Когда клиент обращается к серверу, запрос состоит из:

- HTTP Метод (Глагол): Что мы хотим сделать?
- GET — Получить данные.
- POST — Создать новый ресурс.
- PUT — Заменить ресурс целиком.
- PATCH — Частично обновить ресурс (например, только email).
- DELETE — Удалить ресурс.

- Путь (URL): К какому ресурсу обращаемся? (напр. `/users`, `/posts/25`).
- Заголовки (Headers): Служебная информация (тип контента, токены авторизации).
- Тело (Body): Данные, которые мы отправляем обычно в JSON(POST, PUT, PATCH).

## PATCH vs PUT

- PUT: Вы отправляете объект целиком (Имя, Фамилия, Возраст). Сервер заменяет старый объект новым (in a perfect scenario).

- PATCH: Вы отправляете только { "age": 26 }. Сервер обновляет только это поле, не трогая остальное (in a perfect scenario).

```js
    // "Database"
    {
    id: 5,
    title: "My Post",
    body: "Lorem ipsum"
    }

    // a PUT request should contain everything
    const updated = {
      id: 5,
      title: "My Post -- UPDATED",
      body: "Lorem ipsum"
    }

    // a PATCH request should contain just the updated details
    const patched = {
      title: "My Post -- UPDATED AGAIN"
    }

    fetch("http://localhost:8080/posts/5", { method: "PUT", body: JSON.stringify(updated), headers... })

```

## Коды ответов HTTP (Статус-коды) - Хороший API всегда честно говорит, что произошло:

# 2xx (Успех):

200 OK — Все прошло отлично.

201 Created — Ресурс успешно создан (напр. новый юзер).

# 3xx (Перенаправление):

301 — Ресурс переехал навсегда.

# 4xx (Ошибка клиента):

400 Bad Request — Вы отправили «кривой» JSON или не те данные.

401 Unauthorized — Вы не залогинены.

403 Forbidden — У вас нет прав доступа (напр. попытка удалить чужой пост).

404 Not Found — Страница или ресурс не найдены.

# 5xx (Ошибка сервера):

500 Internal Server Error — У нас на бэкенде что-то упало.

- https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Status

## Архитектура Эндпоинтов (Примеры)

# Правильный RESTful API выглядит логично и иерархично:

- GET /users — Список всех пользователей.
- POST /users — Регистрация нового пользователя.
- GET /users/:id — Данные конкретного пользователя.
- GET /users/:id/todos — Список задач конкретного пользователя.
- DELETE /todos/:id — Удаление конкретной задачи по её уникальному ID.

## Альтернативы REST: Не всё то REST, что API

- SOAP: Олдскульный стандарт. Очень строгий, работает через XML. В 2026 году встречается в основном в старых банковских системах или крупных корпорациях. Сложный для чтения людьми, но надежный для машин.

- GraphQL: Современный и гибкий. Вместо множества путей (URL) у вас только один эндпоинт. Вы сами описываете в теле запроса, какие именно поля вам нужны.

## Инструментарий: Postman

- Postman — это ваш «швейцарский нож». Бэкенд-разработчик не тестирует API через браузер (браузер умеет только GET).
- Позволяет сохранять коллекции запросов.
- Удобно переключаться между методами (POST, PUT, DELETE).
- Показывает красивые JSON-ответы и статусы.

## Порядок в коде: Express Router

- Когда проект растет, файл app.js превращается в кашу. Чтобы этого избежать, мы используем Express Router — он позволяет разбить маршруты по логическим модулям.
